You are working on FRONTIER, a persistent globe-based Algorand strategy game built with:
- React 18 + TypeScript + Vite
- Three.js + @react-three/fiber + @react-three/drei
- Express server
- shared schema types
- current 3D globe with land plots and faction gameplay

Your task is to implement **Phase 1: World Intel / Tactical Replay**.

GOAL
Add a replayable world-history system so the game can record major events and replay them on the globe with a timeline scrubber. This is NOT real-world OSINT. It is purely game-state reconstruction from in-game events.

IMPORTANT RULES
- Do not rewrite the whole globe.
- Do not migrate to ArcGIS or Cesium.
- Keep the current Three.js globe.
- Build this as an additive system.
- Prefer the smallest working implementation first.
- Use full replacement files when needed instead of partial diffs.
- Keep main gameplay working.
- If storage is currently mixed between memory and DB, implement this feature in a way that works now and can be migrated later.

PHASE 1 SCOPE
Implement these features only:

1. WORLD EVENT MODEL
Create a canonical event model for replayable game events.

Add a shared type for:
- id
- type
- timestamp
- endTimestamp optional
- plotId optional
- attackerPlotId optional
- defenderPlotId optional
- lat
- lng
- altitude optional
- factionId optional
- playerId optional
- severity optional
- visibility optional
- metadata object

Supported event types for Phase 1:
- land_claimed
- battle_started
- battle_resolved
- commander_deployed
- scan_ping
- jammer_zone
- faction_movement
- resource_pulse

2. SERVER EVENT STORE
Add a server-side world event store and API endpoints.

If there is already a storage abstraction, integrate into it cleanly.
If there is only in-memory support right now, create a dedicated in-memory event store module that can later be swapped to DB.

Need:
- appendWorldEvent(event)
- listWorldEvents({ start, end, types, limit })
- getRecentWorldEvents()
- optional seedDemoWorldEvents() for local testing

3. AUTO-LOG GAME ACTIONS
Hook event creation into existing gameplay flows wherever practical:
- land purchase / claim -> land_claimed
- attack launch -> battle_started
- attack resolution -> battle_resolved
- commander action / special attack if available -> commander_deployed
- AI faction movement tick if available -> faction_movement
- mining tick or periodic yield pulse -> resource_pulse

If exact hook points are hard to locate, wire the first 3 only:
- land_claimed
- battle_started
- battle_resolved

4. API ENDPOINTS
Add:
- GET /api/world/events
  query params:
  - start
  - end
  - types
  - limit
- GET /api/world/events/recent
- POST /api/world/events/dev-seed
  dev-only endpoint to seed sample replay data

Return normalized JSON.

5. CLIENT TYPES + QUERY HOOK
Add client-side types and query hooks for world events.
Use React Query if already present.

Need:
- useWorldEvents(filters)
- useRecentWorldEvents()

6. TACTICAL REPLAY UI
Create a new client UI panel called something like:
- World Intel
or
- Tactical Replay

This panel should include:
- timeline scrubber
- play/pause button
- speed selector (1x, 5x, 20x)
- layer toggles for:
  - battles
  - claims
  - scans
  - jamming
  - faction movement
- last 15m / 1h / 24h presets

7. GLOBE OVERLAY RENDERING
Render replay events on the existing globe without replacing current land rendering.

Add a lightweight overlay system:
- battle_started -> red/orange pulse at plot
- battle_resolved -> ring burst or resolved marker
- land_claimed -> claim beacon pulse
- scan_ping -> expanding blue scan ring
- jammer_zone -> translucent sphere / halo
- faction_movement -> animated arc or moving point between 2 positions

Keep it lightweight:
- no thousands of heavy meshes
- use memoization
- use instancing or very small mesh counts where possible
- only render filtered events in the active replay window

8. REPLAY TIME FILTERING
Implement client replay state:
- selected timestamp
- playing boolean
- playback speed
- visible event layers
- selected range preset

At time T, render:
- point-in-time events near T
- duration events where timestamp <= T <= endTimestamp

9. MOBILE-FIRST PANEL
Keep the replay controls usable on mobile.
Use existing UI patterns if there is already a bottom sheet / drawer / panel component.

10. SAFE DEFAULTS
If no real events exist, show seeded demo events in dev only.
Do not break current gameplay if replay data fails to load.
